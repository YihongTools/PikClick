# Release v2.1.1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Verify the 0.1–60.0 second delay behavior on the connected Android phone, build a locally signed APK, and publish it as GitHub Release v2.1.1.

**Architecture:** The existing source commit `55f51cf` owns the delay validation and 0.1-second countdown display. The phone is used for black-box verification through `adb`; the release APK is signed only on the maintainer workstation from ignored local keystore files. GitHub receives only the final APK and checksum assets.

**Tech Stack:** Kotlin, Android Gradle Plugin, Gradle Wrapper, JUnit 4, Android SDK `adb`, `apksigner`, GitHub CLI.

## Global Constraints

- Accepted delay range is exactly `0.1–60.0` seconds, with one decimal place in UI and default `3.5` seconds.
- The release private key, `release-key.jks`, and `keystore.properties` must never be committed or uploaded as GitHub Secrets.
- Existing `dist` APK/checksum files are local user changes and must not be staged for this feature.
- Publish a new `v2.1.1` Release; do not overwrite `v2.1.0`.
- A release claim requires fresh local test, signature, checksum, and GitHub asset verification.

### Task 1: Prepare and inspect the connected device

**Files:** None.

- [ ] **Step 1: Locate Android SDK tools.**

Run:

```powershell
$sdkRoot = "$env:LOCALAPPDATA\Android\Sdk"
$adb = Get-ChildItem "$sdkRoot\platform-tools\adb.exe" | Select-Object -First 1
$apksigner = Get-ChildItem "$sdkRoot\build-tools\*\apksigner.bat" | Sort-Object FullName -Descending | Select-Object -First 1
```

Expected: both files exist.

- [ ] **Step 2: Confirm one authorized device.**

Run `& $adb.FullName devices` and confirm exactly one device has state `device`.

- [ ] **Step 3: Capture device facts for the test record.**

Run `& $adb.FullName shell getprop ro.build.version.sdk` and record API level; do not change device auto-rotation settings.

### Task 2: Install and manually verify delay boundaries

**Files:** None.

- [ ] **Step 1: Install the current locally built APK.**

Run `& $adb.FullName install -r '.\dist\PikClick-v2.1.1-release.apk'`. Expected: `Success`.

- [ ] **Step 2: Verify the lower boundary.**

Open PikClick, enter `0.1`, press Done, and confirm the field remains `0.1`; start a sequence and confirm the countdown begins at approximately `0.1s`.

- [ ] **Step 3: Verify the upper boundary.**

Enter `60.0`, press Done, and confirm the field remains `60.0`; start a sequence and confirm countdown is active. Cancel it using the close control.

- [ ] **Step 4: Verify invalid boundaries.**

Enter `0.0`, `0.09`, and `60.1` one at a time. Expected: a range error is shown and the saved valid value is restored; no sequence starts.

- [ ] **Step 5: Verify no automatic rotation change.**

Keep the phone's existing auto-rotation setting unchanged. Rotate the physical device only if the OS setting already permits it; confirm PikClick does not request or toggle rotation and still recalculates the bubble position after a configuration change.

### Task 3: Run automated verification

**Files:** None.

- [ ] **Step 1: Run unit tests, lint, and debug build.**

```powershell
$env:JAVA_HOME = 'C:\Program Files\Microsoft\jdk-17.0.19.10-hotspot'
.\gradlew.bat testDebugUnitTest lintDebug assembleDebug --stacktrace
```

Expected: exit code 0, 6 unit tests, lint issue count 0.

- [ ] **Step 2: Run signed release build and checksum.**

```powershell
.\gradlew.bat clean testDebugUnitTest lintRelease checksumReleaseApk --stacktrace
```

Expected: exit code 0 and `dist/PikClick-v2.1.1-release.apk` plus `.sha256`.

- [ ] **Step 3: Verify checksum and signature.**

Use `Get-FileHash` to compare the APK hash with the `.sha256` file, then run `apksigner verify --verbose --print-certs`. Expected: `Verified` and exactly one signer.

### Task 4: Publish v2.1.1 and verify remote assets

**Files:** None.

- [ ] **Step 1: Create the GitHub Release.**

Use GitHub CLI to create tag `v2.1.1` from current `main`, with release title `PikClick v2.1.1` and notes stating the delay range is now 0.1–60.0 seconds.

- [ ] **Step 2: Upload only the signed APK and checksum.**

Upload `dist/PikClick-v2.1.1-release.apk` and `dist/PikClick-v2.1.1-release.apk.sha256`. Never upload `release-key.jks` or `keystore.properties`.

- [ ] **Step 3: Verify GitHub digests.**

Read the release assets through GitHub CLI and confirm the remote APK digest equals the locally calculated SHA-256 and both assets are `uploaded`.

- [ ] **Step 4: Commit only the plan if required by the workflow.**

Do not stage `dist` binaries. If this plan is committed, use `git add docs/superpowers/plans/2026-07-24-release-v2.1.1-plan.md` and commit it separately from release artifacts.
